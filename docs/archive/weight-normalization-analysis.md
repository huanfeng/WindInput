# 词库权重归一化分析文档

## 1. 现状：各词库的原始权重范围

### 1.1 系统词库

| 词库 | 类型 | sort 模式 | 权重范围 | 中位数 | 条目数 |
|------|------|-----------|----------|--------|--------|
| wubi86_jidian | rime_wubi | by_weight | 5 ~ 890 | 10 | 89,268 |
| base.dict (拼音) | rime_pinyin | — | 0 ~ 19,260,817 | ~200 | 553,130 (不含 tencent) |

**五笔词库权重分布（wubi86_jidian）**：
- weight=10: 77,171 条（86%）— 普通词条
- weight=20: 8,957 条（10%）— 略高频
- weight=30~100: ~2,600 条 — 常用词
- weight=100~890: ~300 条 — 高频字（同编码内排序用）

**拼音词库权重分布（base.dict）**：
- ≤100: 180,924 条（33%）
- 101~1,000: 120,232 条（22%）
- 1,001~10,000: 163,501 条（30%）
- >10,000: 77,268 条（14%）

> **关键差异**：五笔和拼音词库的权重量级差距极大（890 vs 19,260,817），
> 同一个词在两个词库中的权重完全不可比。

### 1.2 用户词库（UserDict）

| 场景 | 权重 | 来源 |
|------|------|------|
| 手动加词（候选框 Enter） | 5000 | `handle_addword.go: addWordMaxWeight` |
| 手动加词（设置界面） | 用户填写，默认 5000 | `AddWordPage.vue: wordWeight` |
| 动态学习新词（首次选中） | 100 | `OnWordSelected(addWeight=100)` |
| 动态提权增量 | +10/次 | `OnWordSelected(boostDelta=10)` |
| 动态提权硬上限 | 2000 | `MaxDynamicWeight` |
| 误选保护阈值 | 选中 2~3 次后才开始提权 | `countThreshold` |
| 仅调频模式 | +10/次 | `IncreaseWeight(delta=10)` |

### 1.3 临时词库（TempDict）

- 与 UserDict 相同的权重体系（继承 MaxDynamicWeight=2000）
- 学习增量：+10/次
- 晋升条件：选中次数 ≥ promoteCount（通常 3~5 次）
- 晋升时：将当前权重迁移到 UserDict

### 1.4 前缀候选降权

五笔引擎对前缀匹配候选做分层降权：
```
prefixCandidates[i].Weight -= remaining * 1000000
```
- 剩余 1 码：-1,000,000
- 剩余 2 码：-2,000,000

混输引擎对五笔侧提权：
```
精确匹配：+10,000,000
前缀匹配：+6,000,000
```

> 这些大数值偏移量只在引擎内部排序时使用，不影响存储。

## 2. 排序链路

### 2.1 CompositeDict 聚合查询

```
层级遍历顺序（LayerType 值小 → 高优先级）：
  Logic(0) → Shadow(1) → User(2) → Temp(3) → Cell(4) → System(5)

去重规则：按 Text 去重，保留首次出现的（优先级高的层）
排序规则：candidate.Better()  →  Weight 降序 → Code 升序 → Text 升序
```

**实际效果**：
- 同一个词在 User 层和 System 层都存在时，User 层的版本被保留（带 User 层的权重）
- 不同词之间的排序**完全由 Weight 决定**

### 2.2 五笔引擎排序流程

```
Phase 1: 精确匹配 → CompositeDict.Search(code)
Phase 2: 前缀匹配 → CompositeDict.SearchPrefix(code)
Phase 3: 前缀降权 → Weight -= remaining * 1000000
Phase 4: 合并 + 去重
Phase 5: 排序（Better / BetterNatural）+ 过滤 + 截断
Phase 6: Shadow 拦截器（pin/delete，不修改 Weight）
```

### 2.3 拼音引擎排序流程

拼音引擎有独立的评分体系，**不直接使用原始 Weight 排序**：

```go
// 1. 归一化：词库权重 → [-15, 0]
NormalizeWeight(dictWeight):
    dictWeight ≤ 0      → -15.0
    dictWeight ≥ 1000000 → 0.0
    其他                 → (dictWeight / 1000000) * 15.0 - 15.0

// 2. Rime 风格评分
score = exp(normalizedWeight) + initialQuality + coverage
    + LM_score * 0.3  // 语言模型加成

// 3. 映射到整数
candidateWeight = int(score * 1000000)
```

**initialQuality 值表**（决定候选来源的基础偏移）：
| 场景 | initialQuality |
|------|---------------|
| 命令候选 | 100.0 |
| Viterbi 造句 | 4.0 |
| 精确匹配（首段） | 4.0 |
| 精确匹配（其他） | 2.0 |
| 简拼词组 | 3.0 |
| 首音节单字 | 2.5~4.0 |
| 前缀匹配 | 2.0 |
| 非首音节单字 | 0.5 |

> 拼音引擎的归一化已经内置（rimeMaxDictWeight=1000000），
> 但 UserDict 的权重（100~2000）在此归一化下会映射到极低值（约 -14.99 ~ -14.97），
> 几乎没有区分度。

## 3. 当前问题分析

### 3.1 五笔模式：用户词权重远超系统词

| 来源 | 权重 | 排序结果 |
|------|------|---------|
| 系统词库（普通） | 10 | 排在所有用户词之后 |
| 系统词库（高频） | 100~890 | 仍排在用户词之后 |
| 用户手动加词 | 5000 | **永远排在所有系统词前面** |
| 动态学习词 | 100~2000 | 初始就排在大部分系统词前面 |

**后果**：
- 用户手动加的任何词都会排在同编码所有系统词前面
- 无法通过权重让用户词"融入"系统词的排序中
- 与 QQ 五笔等输入法的行为不一致（用户词默认排在系统词之后）

### 3.2 拼音模式：UserDict 权重在归一化中被压平

UserDict 的 100~2000 权重在 NormalizeWeight 中：
- 100 → (100/1000000)*15 - 15 = -14.9985
- 2000 → (2000/1000000)*15 - 15 = -14.97

区间宽度仅 0.03，经过 exp() 后差异微乎其微。
相比系统词库 0~19M 的跨度，用户词的权重几乎无区分度。

### 3.3 跨词库不可比

- 五笔词库 weight=10 和拼音词库 weight=10 的含义完全不同
- 用户词库的 weight=100 在五笔中是"高于大部分系统词"，在拼音中是"极低频"
- 无法在混输模式下合理排序来自不同词库的候选

---

## 4. 设计目标

1. **统一内部权重刻度**：定义一个项目统一的权重区间（如 0~10000），所有词库的原始权重映射到此区间
2. **中位对齐**：每个词库标注其中位权重（median），映射到统一刻度的中档位置
3. **用户词合理融入**：
   - 手动加词默认排在同编码系统词的**后面**（权重略低于系统中位）
   - 随着使用频率提升，权重逐渐上升，可以超过系统词
4. **位置固定用 Shadow**：如果用户需要某个词固定在某个位置，使用 Shadow pin 机制
5. **方案级可配置**：不同的 schema 可以配置不同的归一化参数

## 5. 设计方案

基于分析结论和架构评审反馈，确定以下方案。

### 5.1 总体架构：混合处理

| 词库类型 | 归一化时机 | 理由 |
|----------|-----------|------|
| 系统词库（CodeTable/拼音词库） | **wdb 构建时**（编译时） | 数据量大（8.9万~55万条），运行时零开销 |
| 用户词库（UserDict/TempDict） | **写入时 / 查询时** | 数据量小且动态变化，实时转换 |

### 5.2 归一化参数：schema.yaml dictionaries 节

配置位置选择 `schema.yaml` 的 `dictionaries` 节，因为权重分布与具体方案强绑定。

```yaml
dictionaries:
  - id: wubi86_main
    path: "dict/wubi86/wubi86_jidian.dict.yaml"
    type: rime_wubi
    default: true
    weight_spec:
      median: 10        # 中位权重（映射到统一区间的基准点）
      max: 890          # 最大权重
      mode: linear      # 映射算法：linear / log
  - id: pinyin_base
    path: "dict/pinyin/rime_ice.dict.yaml"
    type: rime_pinyin
    weight_spec:
      median: 200
      max: 19260817
      mode: log         # 对数映射（长尾分布）
```

### 5.3 统一区间：0 ~ 10000

```
 0          800    1000         5000              10000
 |           |      |            |                  |
 最低      用户词  系统中位    系统高频词         上限
          (自动   (普通词条    (五笔890/
           学习)   映射到此)    拼音高频)
```

**区间含义**：
- **0**: 最低权重（极罕见词、补充词库 sort=original 的词条）
- **800**: 动态学习词初始权重（低于系统中位，需使用提权）
- **1000**: 系统词库中位基准（五笔 weight=10 映射到此）
- **1200**: 手动加词默认权重（略高于系统中位，低于高频系统词）
- **5000**: 系统高频词区域
- **10000**: 归一化上限（动态权重硬上限也对齐到此）

### 5.4 映射算法

**五笔词库（线性/分段线性映射）**：

五笔词库跨度小（5~890），适合分段线性映射：

```
映射规则：
  原始 median (10)  → 统一 1000
  原始 max (890)    → 统一 5000
  原始 min (5)      → 统一 500

分段公式：
  W_orig ≤ median:
    W_norm = 500 + (W_orig - 5) / (10 - 5) * (1000 - 500)
  W_orig > median:
    W_norm = 1000 + (W_orig - 10) / (890 - 10) * (5000 - 1000)
```

**拼音词库（对数映射）**：

拼音词库跨度极大（0~19M），85% 的词在 10000 以下。线性映射会将大部分词压缩到 0~5，必须用对数：

```
W_norm = ln(W_orig + 1) / ln(W_max + 1) × 10000

示例：
  W_orig = 0          → W_norm = 0
  W_orig = 200        → W_norm = ln(201)/ln(19260818) × 10000 ≈ 3163
  W_orig = 10000      → W_norm = ln(10001)/ln(19260818) × 10000 ≈ 5494
  W_orig = 1000000    → W_norm = ln(1000001)/ln(19260818) × 10000 ≈ 8240
  W_orig = 19260817   → W_norm = 10000
```

### 5.5 用户词权重策略

**核心原则**：尊重肌肉记忆，区分「手动造词」与「自动学习」。

| 场景 | 旧权重 | 新权重（归一化后） | 排序效果 |
|------|--------|-------------------|---------|
| 自动学习（首次） | 100 | 800 | 低于系统中位，排在普通系统词后面 |
| 自动学习（提权增量） | +10/次 | +20/次 | 逐步提升，约 10 次使用后超过系统中位 |
| 自动学习（硬上限） | 2000 | 10000 | 可达最高权重 |
| 手动加词（默认） | 5000 | 1200 | 略高于系统中位，但低于高频系统词 |
| 强制置顶 | — | 不修改权重，使用 Shadow pin | 绕过权重排序，强制指定位置 |

**自动学习提权曲线**（归一化后）：
```
首次选中 → 800, count=1
第2次选中 → 800, count=2  （误选保护，未达阈值）
第3次选中 → 820, count=3  （开始提权，+20）
第4次选中 → 840, count=4
...
第13次选中 → 1020, count=13 ← 此时超过系统中位，开始排在普通系统词前面
...
最终上限 → 10000（需约 463 次超过阈值后的选中）
```

### 5.6 拼音引擎适配

统一刻度后，`NormalizeWeight` 需要同步调整基准：

```go
// 旧版：基准 1000000
const rimeMaxDictWeight = 1000000.0
func NormalizeWeight(dictWeight float64) float64 {
    return (dictWeight / rimeMaxDictWeight) * 15.0 - 15.0
}

// 新版：基准 10000（统一区间上限）
const normalizedMaxWeight = 10000.0
func NormalizeWeight(dictWeight float64) float64 {
    if dictWeight <= 0 { return -15.0 }
    if dictWeight >= normalizedMaxWeight { return 0.0 }
    return (dictWeight / normalizedMaxWeight) * 15.0 - 15.0
}
```

UserDict 权重 800~10000 的归一化效果：
- 800 → (800/10000)*15 - 15 = **-13.8** → exp(-13.8) ≈ 0.000001
- 1200 → (1200/10000)*15 - 15 = **-13.2** → exp(-13.2) ≈ 0.000002
- 5000 → (5000/10000)*15 - 15 = **-7.5** → exp(-7.5) ≈ 0.00055
- 10000 → (10000/10000)*15 - 15 = **0.0** → exp(0) = 1.0

区分度大幅提升（对比旧版 -14.99 ~ -14.97 的 0.02 跨度）。

### 5.7 混输引擎权重平衡

**现状**：混输通过大数偏移量实现跨引擎排序：

```
五笔精确匹配：Weight += 10,000,000
五笔前缀匹配：Weight += 6,000,000
拼音纯辅音简拼 3 码：Weight -= 2,000,000
拼音纯辅音简拼 4 码：Weight -= 3,500,000
```

**归一化后的影响**：
- 五笔引擎输出的候选 Weight 范围：0~10000（归一化后）
- 拼音引擎输出的候选 Weight 范围：int(score × 1000000)，约 0~5,000,000
- 混输偏移量的量级仍然远大于归一化后的五笔权重

**需要同步调整的偏移量**：
归一化后，五笔候选的基础权重从 10~890 变为 500~5000，
混输的 WubiWeightBoost 仍为 10M，量级差异更大但方向一致，功能不受影响。
后续可考虑将混输偏移量也纳入归一化体系，但**不是本次重点**。

## 6. 实施步骤

### 阶段一：基础设施（schema + 数据结构）

1. `schema.go` 新增 `WeightSpec` 结构体
2. `schema.yaml` 各方案添加 `weight_spec` 配置
3. `schema/loader.go` 解析 `weight_spec`

### 阶段二：WDB 构建时归一化

4. `dictcache/convert.go` 中 `ConvertRimeWubiToWdb` 和 `ConvertPinyinToWdb`
   加入映射逻辑（根据 `weight_spec.mode` 选择线性/对数算法）
5. 重建所有 wdb 文件

### 阶段三：用户词库适配

6. 调整 `UserDict` 常量：`MaxDynamicWeight` → 10000
7. 调整 `handle_addword.go`：`addWordMaxWeight` → 1200
8. 调整 `OnWordSelected` 参数：`addWeight` → 800, `boostDelta` → 20
9. `AddWordPage.vue` 默认权重调整为 1200，范围改为 0~10000

### 阶段四：拼音引擎适配

10. `ranker.go` 修改 `NormalizeWeight` 基准为 10000
11. 验证拼音候选排序效果

### 阶段五：验证与回归

12. 五笔模式：验证用户词排在系统中位之后、手动加词排在合理位置
13. 拼音模式：验证 UserDict 权重区分度
14. 混输模式：验证跨引擎排序不受影响
15. 已有用户词库的兼容性处理（旧权重到新权重的迁移）
