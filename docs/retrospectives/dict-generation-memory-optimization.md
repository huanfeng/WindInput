# 复盘：低内存环境下词库生成的内存优化

## 背景与交付

输入法首次安装时需把 rime 文本词库转换为二进制 `wdat`/`wdb`。实测拼音词库（524903 条全拼 + 184496 条简拼）转换的堆峰值可达数百 MB，在可用物理内存不足的机器上会触发 Go runtime `cannot allocate memory` 而崩溃——`crash.log` 中可见一次 OOM 精确发生在 `DATBuilder.Build` 的 `makeslice`（旧 map 版 `dat.go:212`），崩溃进程仅存活 3.6 秒、堆约 700MB 时系统即拒绝分配。

交付三层优化（均不改变输出文件内容，仅降内存峰值）：

| 层级 | 改动 | 性质 |
|------|------|------|
| 一 | `datformat/dat.go`：DAT 构建期 trie 节点 `children` 从 `map[byte]*trieNode` 改为按字节升序的 `[]trieChild`（二分查找/插入） | **无条件生效**，主要功臣 |
| 二 | `binformat`/`datformat` 两个 writer 新增 `SetLowMemory()`；`dictcache/convert.go` 低内存时边转换边 `delete` 源 map + 主动 GC | 低内存模式 |
| 三 | 新增 `pkg/sysinfo`：`GlobalMemoryStatusEx` 读可用物理内存，`LowMemoryMode()` 在可用 < 1024MB 时启用省内存路径 | 决策入口 |

效果：端到端转换峰值约 481MB（快速）/462MB（低内存）。生产日志验证——可用内存 512MB 的真实运行下，低内存模式正确触发并成功完成全部词库生成，未再 OOM。

**层级四（DAT.Build 内 trie 增量释放）经实测否决并回退**，见下文「关键教训」。

相关代码：`internal/dict/datformat/dat.go`、`internal/dict/datformat/writer.go`、`internal/dict/binformat/writer.go`、`internal/dict/dictcache/convert.go`、`pkg/sysinfo/`。

## 流程概览

1. **定位内存炸弹**：通读 wdat/wdb 生成链路，发现峰值最大单一来源不是词条数据本身，而是 DAT 构建期每个 trie 节点携带的空 `map[byte]*trieNode`——数百万节点 × ~128B map header ≈ 数百 MB，纯结构开销。
2. **三层落地**：trie 结构瘦身（层级一）+ 阶段间释放（层级二）+ 内存检测（层级三）。每层独立可验证，编译/格式化/单测全绿。
3. **独立二次审查**：用 reviewer 子代理审查四个层级的逻辑正确性，配合逐行论证（DAT 输出逐字节一致性）+ 真实往返测试。
4. **日志取证**：用户提供运行日志后，用「栈帧行号 + git 版本比对」确认 OOM 属于优化前旧二进制；并把每次拼音 wdat 转换的「开始/低内存/完成」配对，确认优化后所有转换均成功完成。
5. **数据驱动评估层级四**：用本机真实词库实测峰值构成，尝试 trie 增量释放，**实测峰值不降反升**，回退。

## 关键数据（本机真实词库，524903 codes）

主 DAT 构建内存分解：

| 成分 | 大小 |
|------|------|
| trie（层级一后的有序切片版） | 107.5 MB |
| base+check（`int32×2`，DAT.Size=2714762） | 按 size 20.7 MB / 按 cap(2²²) 32.0 MB（cap/size=1.54x，尾部空洞浪费 11.3 MB） |
| 主 DAT Build 峰值 | 221 MB |

端到端 `ConvertPinyinToWdat` 峰值：

| 路径 | 峰值 HeapInuse | 输出 |
|------|---------------|------|
| 快速 | 481 MB | 47.6 MB |
| 低内存 | 462 MB | 47.6 MB |

> 注意：低内存路径相比快速路径仅削峰约 19MB——层级二释放的 `Entries` 被紧随其后的 `DAT.Build` 立刻吃掉，两个高峰首尾相接，所以「先释放再分配」几乎不降低**瞬时**峰值。峰值是输出体积的 ~10 倍。

## 关键教训

### 1. 内存炸弹常在「结构开销」而非「数据本身」
百万级 trie 节点的空 `map` header（~128B/个）是峰值主因，远超候选词文本。换成有序切片（拼音字符集仅 26 字母，二分成本可忽略）即削掉数百 MB，且对所有用户无条件生效、无需任何内存判断。**改造前先量化各结构的实际占用，不要凭"数据量大"想当然。**

### 2. 「主动置 nil 制造垃圾」在分配密集循环里会抬高峰值（反直觉）
层级四尝试在 BFS 中边遍历边 `node.children = nil` 释放已处理子树，期望削峰。实测相反：

| 指标 | 改前 | 改后 |
|------|------|------|
| 主 DAT Build 峰值 | 221 MB | **276 MB（+55）** |
| 端到端快速路径 | 481 MB | **521 MB（+40）** |

原因：`children=nil` 产生的是**垃圾**而非即时回收，GC 跑到之前「旧 children 垃圾 + 新增 base/check」同时计入 `HeapInuse`；而原版 children 一直存活、heap 稳定。**`HeapInuse` 统计的是"已分配未还给 OS 的 span"，垃圾在被回收前照样占着。要降峰必须减少"同时存活"的对象，而不是更早把对象变成垃圾。** trie 是 DA 构建的固有中间产物，只要算法仍"先建完整 trie 再转换"，这 107MB 就降不掉。

### 3. 「先释放再分配」削不动首尾相接的峰值
层级二把释放点放在 `DAT.Build` 之前，但 Build 立刻把腾出的空间吃回去，瞬时峰值几乎不变（仅 -19MB）。真正的削峰需要错开两个高峰，或减少峰值时刻的存活集。

### 4. 日志取证：栈帧行号 + git 版本比对能精确定位崩溃属于哪个二进制
`crash.log` 累积多天多次崩溃。通过 `dat.go:212` 在优化前是 `make`（会 OOM 的分配点）、优化后是 `return`（不分配），一行之差就把 OOM 钉死在「优化前 map 版」。配合「转换开始/完成」配对，确认优化后无一次中断。

## 当前结论与后续方向

- **层级一~三是合理终点**。端到端峰值 ~481MB，512MB 可用内存场景已被生产日志验证可跑通。层级四不可行（数据为证）。
- **若需支持比 512MB 更苛刻的可用内存**：治本之道不是继续抠 Build 内存（trie 是固有成本），而是**架构隔离**——把拼音 wdat 生成放进独立子进程（生成完即退出，内存完全归还 OS）。项目已有 `cmd/gen_bindict`、`cmd/gen_codetable_wdb` 生成器入口与 `internal/proc` 进程管理可复用。好处：① 生成峰值与输入法主进程彻底隔离；② 子进程 OOM 不带崩主进程。属于架构调整，需 IPC + 入口，作为独立任务评估。

## 调试开关

- `WINDINPUT_FORCE_LOWMEM=1`：强制启用省内存路径（`0`/`off` 强制关闭），用于测试/手动指定。
- `WINDINPUT_LOWMEM_MB=<N>`：覆盖低内存判定阈值（默认 1024MB）。
