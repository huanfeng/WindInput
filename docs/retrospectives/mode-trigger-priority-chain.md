# 复盘：模式激活键优先级回落链

## 背景与交付

把输入法的「模式激活键 vs 候选选择键」从**互斥门禁**重构为**优先级回落链**，并让所有触发键激活的模式（快捷输入 / 临时拼音 / 临时英文）支持「正在输入时按触发键 → 顶码上屏当前高亮候选 → 进入模式」，触发键字符本身不上屏。

交付：新增 `internal/coordinator/mode_trigger.go` 核心 + 5 个相关文件改动 + 10 个单元测试 + 设计文档，全量测试（36 包）0 失败。

相关文档：`docs/design/mode-trigger-priority-chain.md`（设计）、`...-plan.md`（实现计划）。

---

## 流程概览

1. **需求分析**：先调研现有代码（临时拼音/快捷输入/临时英文三套同构的触发判定），再逐项确认边界（空码处理、是否依赖 `punct_commit`、适用范围、提交哪个候选、模式间优先级）。
2. **需求演进**：初版只针对临时拼音；澄清后扩展为「所有触发键模式统一」+「二三候选键优先级高于模式激活键」+「候选不足时回落」。设计随之从「单点补丁」升级为「统一优先级回落链」。
3. **设计**：对比三种实现形态（集中回落链+轻量模式表 / 完整接口注册表 / 渐进补丁），选中第一种——优先级集中可读、可单测、易扩展。
4. **实现**：拆成 7 个小任务，逐个 TDD（先写失败测试 → 实现 → 验证 → 下一个），每步以编译+测试为完成判据。
5. **中途纠偏**：逐任务审查时发现一处回归隐患（见下），及时修正实现计划。
6. **收尾**：单一提交并入主线（fast-forward）、同步更新目录 `AGENTS.md`。

---

## 做得好的（值得保留）

1. **需求边界逐项钉死，而非假设**。空码、`punct_commit` 依赖、适用范围、提交哪个候选、模式间顺序——每个都先确认再动手，避免「按猜测实现」导致返工。

2. **先找现成范本再设计**。发现已有的「五笔顶码上屏」路径与本需求同构，以及「选词上屏」与「提交后重启编排」两块现成能力，把「发明新机制」降级为「组装已有能力」，风险面大幅收窄。

3. **决策与执行分离 —— 本次最成功的设计选择**。把「优先级裁决」抽成一个纯函数（只读状态、无副作用），让优先级链的全部分支用单元测试覆盖、完全不依赖 UI/引擎/系统交互。这是防回归的真正支点。

4. **以编译+测试为唯一完成判据**。每个任务都亲自跑 `go build` + `go test` 确认，不以「写完了」为准。多次发现工具的诊断缓存给出过时的报错，靠实测才确认真实状态。

5. **逐任务审查抓到回归隐患**。见下条。

## 问题与教训

### 1. 「被排除的特殊项」生命周期分析滞后（z 键回归隐患）

设计阶段决定「`z` 键不走新的回落链、改走独立路径」，但只确认了 `z` 的**回退**路径，漏了 `z` 的**首次触发**路径。隐患潜伏到实现末期才被逐任务审查抓到：原计划要删除的旧触发函数里，恰恰藏着 `z` 的首次触发逻辑。

**教训**：当设计决定「把某个特殊 case 排除、走独立路径」时，必须**逐职责确认那条独立路径覆盖了原 case 的全部职责**。`z` 有「首次触发」和「回退」两个职责，只确认了一个就埋下了回归。这类分析应在设计阶段完成，而非留到实现末期撞上。

### 2. 审查若无书面结论，等于没做

最终的整体代码审查没有留下可追溯的书面结论，只能退回到「作者视角自审」。而自审天然有盲区——作者看不到自己的思维定式。结果是：功能在真机验证前，**没有经过真正独立的第二双眼**，真机测试成了唯一的第二道关。

**教训**：代码审查必须产出**可追溯的书面结论**（file:line 级的发现清单），否则审查环节形同虚设；且独立审查不可被作者自审替代。关键功能上应保证「作者之外的视角 + 书面记录」二者齐备。

### 3. 需求演进未被预判

第一轮只谈临时拼音，第二轮才扩展到全模式 + 优先级，导致一轮设计返工。若在最初就主动问一句「这个行为是只针对临时拼音，还是所有同类模式都要统一？」，可能一次到位。

**判据**（避免走向过度设计的另一极）：当需求里的概念明显有**同类兄弟**（临时拼音 ↔ 快捷输入 / 临时英文）时，值得主动探测泛化意图；孤立概念则不必预先泛化。

### 4. 脏工作区的 git 纪律

整个改动期间，工作区一直存在 4 个与本任务无关的外部未提交文件，且暂存区中途被意外清空过一次。每次提交都必须精确规避这些外部改动。

**教训**：工作区有外部未提交改动时，**坚持用精确文件路径 `git add`，绝不用 `git add -A/.`**；每次提交前重新核对暂存清单（`git diff --cached --name-only`）。本次做到了，应固化为纪律。

### 5. 执行层缺单测

优先级裁决（决策层）单测覆盖很好，但「提交候选 + 重启编排」的执行层只靠真机验证，是测试盲区。

**改进**：可对底层依赖做轻量替身，补一个执行层「状态变更顺序」的单测，把「只能真机验」的范围再缩小。

## 可直接落地的改进清单

1. **设计阶段做「排除项全职责确认」**：任何「走独立路径」的特殊 case，逐职责确认独立路径全覆盖，不留到实现末期。
2. **审查必须落成书面结论**：file:line 级发现清单可追溯；独立审查不被作者自审替代。
3. **需求分析增加「泛化探测」**：概念有同类兄弟时，主动问范围。
4. **以编译+测试为完成判据**：不信「写完了」，每步亲验；警惕诊断缓存的过时报错。
5. **脏工作区纪律**：精确路径 `add`，提交前核对暂存清单。
6. **复杂副作用环境里的路由/优先级逻辑 → 先抽纯决策函数**（本次成功模式，推广）。

---

## 沉淀的可复用模式

**「决策 / 执行分离」用于可测的路由逻辑**：

当一段逻辑要在「充满副作用的环境」（UI、引擎、系统交互）里做优先级或路由裁决时，先把「裁决」抽成一个纯函数——输入是只读状态，输出是一个描述「该做什么」的数据结构；再由一个薄薄的执行层把数据结构翻译成副作用。

收益：裁决逻辑（最容易回归的地方）可以用单元测试穷举所有分支，完全不依赖难以构造的运行环境；执行层保持简单、一一映射。本次的优先级回落链正是靠这个结构，把「最易回归的优先级裁决」钉死在单测里。

## 一句话总结

设计与实现质量高（决策/执行分离 + 复用现成机制 + 每步实测），审查也确实做了并抓到了真实隐患；最大短板是**被排除特殊项（z）的生命周期分析滞后到实现末期**，以及**一次独立审查的结论交付不可靠、价值流失**。下次重点：把「排除项全职责确认」前移到设计阶段，并为重要审查建立可靠的书面结论留存机制。
